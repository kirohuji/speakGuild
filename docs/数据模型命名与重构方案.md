# 数据模型命名与重构方案

> 目标：统一学习包、日常练习、故事剧情、雅思考试三类内容的主干结构；清理遗留表；把"剧本系统"从独立概念收敛到学习包体系中。
> 
> ⚠️ **以下内容为历史提案，所提改动均已实施。请以 `schema.prisma` 为实际参考。**

## 1. 核心判断

当前系统已经形成了一个清晰主干：

```text
LearningPackage
  -> Scene
    -> TrainingTopic
    -> GameMap / GameLocation / GameCharacter
```

现阶段的“学习包”应继续理解为：

```text
一个学习包 = 一个 Scene = 一组训练话题 + 一个实战地图
```

学习包需要拆成两个维度：

```text
一级类型 packageType：内容形态 / 产品形态
  daily       日常练习
  exam        考试专项
  story       故事剧情
  course      课程/付费课
  foundation  零基础

二级分类 SceneCategory：内容主题 / 题材分类
  留学生活、校园生活、雅思口语、历史文化、寓言故事、职场交流...
```

这里需要明确：`GameMap / GameLocation / GameCharacter` 不是 daily 专属表，而是通用“世界层”。

```text
daily:
  练习后进入同一 Scene 的地图做实战自由探索

story:
  可先进入 StoryEpisode 做剧情 play，也可进入同一故事世界地图做自由探索/支线对话

exam:
  可以没有地图，也可以用“考场流程地图”表达候考室、Part 1、Part 2、Part 3
```

所以应统一的是外层和世界层：

```text
LearningPackage / Scene / GameMap / GameLocation / GameCharacter
Chunk / Vocabulary / SentencePattern / ExpressionItem
```

应保留差异的是 play/attempt 记录：

```text
daily: PracticeSession / PracticeTurn
story: StoryEpisode / StoryTurn / StoryRecord
ielts: 先复用 PracticeSession，后续必要时新增 ExamAttempt
```

## 2. 学习包类型建议

给 `LearningPackage` 增加类型字段：

```prisma
enum LearningPackageType {
  daily
  exam
  story
  course
  foundation
}

model LearningPackage {
  ...
  type LearningPackageType @default(daily)
}
```

页面和接口基于 `LearningPackage.type` 分流：

```text
daily:
  学习包详情展示训练话题 + 实战地图

story:
  学习包详情展示剧情章节/剧集 + 可选故事地图/自由探索

exam:
  学习包详情展示考试题型、专项话题模块、模拟考试入口；具体考试如 IELTS 放到 topic metadata
```

## 3. 剧情系统命名统一

现在的 `ScriptEpisode` 实际上承载的是“故事剧情 play 关卡”，不是普通脚本。建议统一改名为 `StoryEpisode`。

### 3.1 表名改名

| 当前名称 | 建议名称 | 理由 |
|---|---|---|
| `ScriptEpisode` | `StoryEpisode` | 对应产品概念“故事/剧情模式”，避免和 Ink script 混淆 |
| `ScriptDialogue` | `StoryTurn` | 和 `PracticeTurn` 对齐，表示一次对话轮次 |
| `ScriptRecord` | `StoryRecord` | 表示用户对某个剧情关卡的完成记录 |
| `ScriptEpisodeVocab` | `StoryEpisodeVocabulary` | 命名完整，避免 `Vocab` 缩写 |
| `ScriptEpisodeChunk` | `StoryEpisodeChunk` | 可保留 Chunk 术语 |
| `ScriptEpisodeSentencePattern` | `StoryEpisodeSentencePattern` | 与 `TrainingTopicSentencePattern` 对齐 |

### 3.2 字段改名

`ScriptEpisode` 当前字段有一部分是“剧情关卡字段”，一部分是“学习包/练习字段”。建议改名如下：

| 当前字段 | 建议字段 | 说明 |
|---|---|---|
| `chapterId` | `chapterKey` | 避免误以为是外键；如果后续有 Chapter 表，再用 `chapterId` |
| `chapterTitle` | `chapterName` | 与 `title` 区分，表示展示名 |
| `episodeOrder` | `sortOrder` | 全项目统一排序字段名 |
| `requiredOutputLevel` | 保留 | 与 Scene/Map 一致 |
| `requiredUserLevel` | 保留 | 与 Scene 一致 |
| `vocabRequiredCount` | `requiredVocabularyCount` | 不用缩写 |
| `vocabTotalCount` | `totalVocabularyCount` | 不用缩写 |
| `chunkRequiredCount` | `requiredChunkCount` | 与上面一致 |
| `chunkTotalCount` | `totalChunkCount` | 与上面一致 |
| `prerequisiteEpisodes` | `prerequisiteEpisodeIds` | 明确存的是 id 列表 |
| `objectives` | `goals` 或 `objectives` | 二选一；建议保留 `objectives`，和 AI prompt 更贴近 |
| `passObjectiveCount` | `requiredObjectiveCount` | 表示通关规则 |
| `passChunkCount` | `requiredUsedChunkCount` | 表示通关规则 |
| `passRetellRequired` | `requiresRetell` | 布尔字段统一用动词 |
| `passMinDialogues` | `minimumTurnCount` | 与 turn 术语对齐 |
| `npcName` | `characterName` | 剧情不一定是 NPC，统一为 character |
| `npcRole` | `characterRole` | 同上 |
| `npcPersonality` | `characterPersona` | 更贴近 prompt 语义 |
| `inkScriptId` | 保留 | 指向 InkScript |
| `coreVocabularies` | `vocabularies` | 不需要 core；关系表已经表达 episode 绑定 |
| `coreChunks` | `chunks` | 不需要 core |
| `corePatterns` | `sentencePatterns` | 和全局模型一致 |
| `dialogues` | `turns` | 和 `PracticeTurn` 对齐 |
| `records` | `records` | 可保留 |
| `isPreview` | 保留 | 与地图/地点 preview 语义一致 |

重点：`coreVocabularies/coreChunks/corePatterns` 建议改掉。  
“核心”是业务展示文案，不应进入关系字段名；在表关系里，挂到剧情关卡上的词汇/句块天然就是该关卡的目标内容。

建议目标结构：

```prisma
model StoryEpisode {
  id                      String   @id @default(cuid())
  scene                   Scene    @relation(fields: [sceneId], references: [id])
  sceneId                 String
  chapterKey              String
  chapterName             String
  sortOrder               Int
  title                   String
  description             String?  @db.Text
  requiredOutputLevel     String   @default("L2")
  requiredUserLevel       Int      @default(1)
  requiredVocabularyCount Int      @default(6)
  totalVocabularyCount    Int      @default(10)
  requiredChunkCount      Int      @default(6)
  totalChunkCount         Int      @default(10)
  prerequisiteEpisodeIds  String[] @default([])
  objectives              String[]
  requiredObjectiveCount  Int      @default(3)
  requiredUsedChunkCount  Int      @default(3)
  requiresRetell          Boolean  @default(true)
  minimumTurnCount        Int      @default(3)
  rewards                 Json?
  characterName           String
  characterRole           String
  characterPersona        String?
  inkScriptId             String?
  vocabularies            StoryEpisodeVocabulary[]
  chunks                  StoryEpisodeChunk[]
  sentencePatterns        StoryEpisodeSentencePattern[]
  turns                   StoryTurn[]
  records                 StoryRecord[]
  isPreview               Boolean  @default(false)
  createdAt               DateTime @default(now())

  @@index([sceneId, chapterKey, sortOrder])
  @@map("story_episode")
}
```

当前已落地：

- Prisma 模型已从 `ScriptEpisode/ScriptDialogue/ScriptRecord` 改为 `StoryEpisode/StoryTurn/StoryRecord`。
- 关系字段已统一为 `vocabularies/chunks/sentencePatterns/turns/records`。
- 业务字段已统一为 `chapterKey/chapterName/sortOrder/characterName/characterRole/characterPersona` 等新名称。
- 为降低迁移风险，物理表暂时通过 `@@map/@map` 继续复用原 `script_*` 表和旧列名；这是兼容层，不是新的产品命名。
- 后端 service/controller 已改用 Story 模型；外部 `/script` 路由和部分返回字段暂时保留旧别名，保证现有移动端页面不断。

## 5. 雅思类型建议

雅思类学习包本质仍是话题练习，建议先复用 `TrainingTopic`，不要立刻新增大表。

给 `TrainingTopic` 增加：

```prisma
enum TrainingTopicType {
  daily
  ielts
}

model TrainingTopic {
  ...
  type     TrainingTopicType @default(daily)
  metadata Json?
}
```

IELTS metadata 示例：

```json
{
  "exam": "IELTS",
  "section": "speaking",
  "part": 2,
  "bandTarget": "6.5",
  "questionType": "cue_card",
  "prepSeconds": 60,
  "answerSeconds": 120,
  "rubric": ["fluency", "lexical_resource", "grammar", "pronunciation"]
```

```text
ExamAttempt / ExamTurn / ExamScore
```

现阶段不建议过早拆。

## 6. 成就系统清理

当前存在两套成就：

```text

  UserAchievementV2

  /achievements
```

实际使用判断：

- 前台 `AchievementHallPage` 使用 `/achievements`，即 V2。
- 后台成就管理使用 `AchievementDef`，即 V2。
- V1 只剩 legacy controller 和旧的 `AchievementPage` API。
- `seed.ts` 仍然清理 `achievement/user_achievement`，但主 seed 使用的是 V2 的 `AchievementDef`。

建议：

### 6.1 保留

```text
AchievementDef
UserAchievementV2
AchievementEngineService
AchievementHallPage
AdminAchievementsPage
```

### 6.2 删除或迁移后删除

```text
Achievement
UserAchievement

GET /achievements/legacy/mine
frontend/features/achievement/api.ts
```

### 6.3 命名优化

| 当前名称 | 最终名称 |
|---|---|
| `AchievementDef` | `Achievement` |
| `UserAchievementV2` | `UserAchievement` |

迁移顺序：

1. 删除 V1 前端页面和 API。

当前已落地：

## 7. 其他疑似冗余或需收敛项

### 7.1 `sentencePatternsDeprecated`

`TrainingTopic.sentencePatternsDeprecated Json? @map("sentence_patterns")` 已经被 `TrainingTopicSentencePattern` 替代。建议确认 seed 和后台没有再写入后删除。

### 7.2 `ScriptEpisode` 相关字段计数

`vocabRequiredCount/vocabTotalCount/chunkRequiredCount/chunkTotalCount` 与 join 表数量存在重复。

建议保留 required count，因为它是解锁/通关规则；total count 可以由 join 表计算。

推荐：

```text
保留:
  requiredVocabularyCount
  requiredChunkCount

删除或改为派生:
  totalVocabularyCount
  totalChunkCount
```

如果后台需要展示“总数”，从关系数量 `_count` 或 include 长度计算。

### 7.3 `InkScript` 多余外键字段

`InkScript` 当前有：

```text
episodeId
locationId
characterId
topicId
trainingTopic relation
```

但 `TrainingTopic` 已有 `inkScriptId`，`GameLocation` 有 `inkScriptId`，`GameLocationNpc` 有 `inkTalkScriptId`，`ScriptEpisode` 也有 `inkScriptId`。

建议统一方向：由内容实体持有 `inkScriptId`，`InkScript` 不反向保存各种 `xxxId` 字段。

可保留：

```text
InkScript.key
InkScript.title
InkScript.scriptType
InkScript.inkJson
InkScript.inkSource
InkScript.declaredVariables
```

迁移后考虑删除：

```text
InkScript.episodeId
InkScript.locationId
InkScript.characterId
InkScript.topicId
```

### 7.4 `UserSceneProgress.completedScriptCount`

如果 story 纳入学习包体系，`completedScriptCount` 需要重新定义。

建议：

```text
daily scene progress:
  completedPracticeCount
  mapUnlocked / mapUnlockedAt

story progress:
  completedStoryEpisodeCount
  completedStoryExplorationCount
  放到 StoryRecord、ExplorationRecord 或 PackageProgress
```

后续可以新增统一进度表：

```prisma
model UserLearningPackageProgress {
  userId
  packageId
  completionPercent
  completedActivityCount
  totalActivityCount
  currentActivityId
  updatedAt
}
```

## 8. 推荐迁移步骤

### Phase 2：故事系统收敛命名

## 9. 最终目标 Tree

```text
LearningPackage(type=daily)
  -> Scene
    -> TrainingTopic(type=daily)
      -> vocabularies
      -> chunks
      -> sentencePatterns
      -> inkScript
      -> PracticeSession
    -> GameMap
      -> GameLocation
      -> GameCharacter
      -> ExplorationRecord

LearningPackage(type=story)
  -> Scene
    -> GameMap
      -> GameLocation
      -> GameCharacter
      -> ExplorationRecord
    -> StoryEpisode
      -> vocabularies
      -> chunks
      -> sentencePatterns
      -> inkScript
      -> StoryTurn
      -> StoryRecord

LearningPackage(type=exam, category=雅思口语)
  -> Scene
    -> TrainingTopic(type=ielts, metadata.exam=IELTS)
      -> vocabularies
      -> chunks
      -> sentencePatterns
      -> PracticeSession
```

核心原则：

```text
外层统一：LearningPackage / Scene
内容资产统一：Vocabulary / Chunk / SentencePattern / InkScript
玩法记录分开：PracticeSession / StoryRecord / ExamAttempt
字段命名统一：vocabularies / chunks / sentencePatterns / turns / records
```
