# 移动端性能体检与优化建议

> 最后更新：2026-07-05
> 范围：`apps/frontend` 的 Capacitor 移动端体验、首包、运行时热路径、离线学习包、本地 AI/STT、代码冗余。

## 一、结论

当前移动端架构已经做了几件正确的事：

- 路由层已经有懒加载，后台管理、练习会话、表达库、成就、排行榜、邀请页、系统文档页都不是首屏同步加载。
- Capacitor 启动重任务已经延迟：学习包检查约 10s 后执行，OTA 检查约 15s 后执行。
- 键盘插件、分享、剪贴板、学习提醒等部分原生能力已经按需动态导入。
- 学习包下载/卸载已任务化，退后台会标记暂停，避免用户以为卡死。
- 音频播放优先本地缓存，缺失时通过 `audioAssetId` private URL 兜底。

主要风险不在单个页面慢，而在“移动端首包和运行时重资源边界”：

- 当前 `dist` 约 36MB，其中 `ort-wasm-simd-threaded.asyncify` 约 22MB。
- 主入口 JS 约 2.6MB，admin chunk 约 2.4MB。
- `sql-wasm` 存在两份：`sql-wasm-1.11.0.wasm` 和 `sql-wasm.wasm`，各约 640KB。
- `logo.png` 约 964KB，`yyzz.png` 约 3.8MB，如果被打进移动端包，需要确认是否必要。
- RevenueCat、语音输入、本地模型管理、profile 设置页存在可继续拆分或抽象的空间。

## 二、现状指标

本次静态检查结果：

```text
apps/frontend/src: 约 4.2MB
apps/frontend/dist: 约 36MB
前端 TS/TSX 文件数: 181
```

`dist` 中较大的文件：

| 文件 | 大小 | 说明 |
|------|-----:|------|
| `assets/ort-wasm-simd-threaded.asyncify-*.wasm` | ~22MB | ONNX Runtime / Transformers.js 运行时，和本地 AI/STT 相关。 |
| `assets/index-*.js` | ~2.6MB | 主入口包。 |
| `assets/admin-routes-*.js` | ~2.4MB | 后台管理路由包，移动端不注册，但如果在同一 build 中仍会产出。 |
| `yyzz.png` | ~3.8MB | 静态图片资源，需要确认移动端是否需要。 |
| `logo.png` | ~964KB | 静态图片资源，建议压缩或替换尺寸。 |
| `sql-wasm*.wasm` | ~640KB × 2 | Web SQLite fallback，疑似重复产物。 |
| `warmup-embedding.worker-*.js` | ~524KB | 本地 warmup judge worker。 |
| `ink-engine-*.js` | ~372KB | VN / Ink 引擎相关。 |

这些数字不等于线上真实下载量，但对 Capacitor 包体、首启解压和 WebView 解析都有参考意义。

## 三、已经做对的优化

### 1. 启动时序

`NativeBridgeProvider` 已经避免把所有重任务压在首帧：

- `SplashScreen.hide()`
- `StatusBar.setStyle()`
- `CapacitorUpdater.notifyAppReady()`
- `RevenueCat.configure()`
- 学习包检查延迟执行
- OTA 检查延迟执行
- resume 检查有 5 分钟节流

这条策略是正确的。后续优化重点不是“再加更多启动任务”，而是减少首屏静态 import 和原生插件同步加载。

### 2. 路由懒加载

`App.tsx` 已经对非首屏页面使用 `lazy()`：

- admin routes
- practice session
- expression library
- achievement hall
- leaderboard
- invite
- portal/company/system pages

这能有效避免 admin 和重页面进入首屏主包。

### 3. 离线学习包

学习包安装已从“不可见长任务”变成任务 drawer：

- 任务状态可见。
- 退后台可暂停。
- 继续下载/卸载可从 drawer 或卡片触发。
- 进度只允许前进，避免 99% 回退。

这对用户感知性能比单纯加快下载更重要。

## 四、高优先级优化

### P0. 移动端专用 build，排除后台和官网资源

当前同一份 frontend build 会产出 admin chunk、官网页、部分后台依赖。虽然移动端路由不注册 `/admin/*`，但产物仍进入 `dist`，如果直接 `cap copy`，这些文件会进入 App 包。

建议新增移动端构建模式：

```text
pnpm --filter @manyu/frontend build:mobile
```

目标：

- `VITE_TARGET=mobile`
- 不生成 admin routes。
- 不生成 portal/company/marketing 这类 Web 独有页面。
- 大型后台依赖不进入移动端 `dist`。

实现方向：

- 拆 `App.mobile.tsx` / `App.web.tsx`，或在 `vite.config.ts` 用 alias 指向不同 routes。
- admin 入口只在 web build 中引用。
- 检查 `@remotion/player`、`pixi.js` 后台编辑器、markdown editor、table 等依赖是否被移动端主图引用。

收益：

- 降低 App 包体。
- 降低 WebView 首次扫描/解压成本。
- 降低 OTA 包大小。

### P0. ONNX Runtime wasm 不应默认进入移动端基础包

`ort-wasm-simd-threaded.asyncify` 约 22MB，是目前最大单文件。它来自 Transformers.js / ONNX runtime。

如果本地 warmup judge 或本地 STT 是可选能力，不建议默认随 App 基础包分发全部 runtime 和模型相关资源。

建议：

- 本地 AI worker 继续懒加载。
- 检查 `@huggingface/transformers` 是否只在 worker 内被引用。
- 尝试将 ONNX wasm 配置为远程按需加载，或作为“本地 AI 模型包”的一部分由用户下载。
- 如果 iOS 审核或离线能力要求必须内置，则只内置 warmup judge 所需最小 runtime，不把 STT 相关 runtime 默认带上。

验收标准：

- 关闭本地 AI/STT 时，移动端 `dist` 不包含 22MB ONNX wasm，或至少不进入首包。
- 首次打开首页不加载 ONNX runtime。

### P0. 首屏静态页面继续拆分

`App.tsx` 当前仍静态导入多个用户端页面：

- `LearningPlanPage`
- `TodayTaskPage`
- `LearningUnitPage`
- `ProfilePage`
- `AccountPage`
- `MemberPage`
- `NotificationListPage`
- `NotificationDetailPage`
- `FeedbackPage`

这些页面不都是首屏必需。移动端真正首屏通常是 `/`、登录页、以及底部 tab 的少数页面。

建议：

- 保留 `/`、`/auth/login`、`/auth/register` 静态。
- 将 `/learning/units/:unitId`、`/account`、`/member`、`/notifications`、`/feedback` 改为懒加载。
- 如果底部 tab 切换体验担心卡顿，可在首页首帧后 idle 预加载 `/learning`、`/today`、`/profile`。

收益：

- 降低主入口 JS。
- 首屏 parse/compile 更稳。

## 五、中优先级优化

### P1. RevenueCat 改为按需动态导入

`lib/native/revenuecat.ts` 顶部静态导入：

```ts
@revenuecat/purchases-capacitor
@revenuecat/purchases-capacitor-ui
```

并且 `NativeBridgeProvider` 静态导入 `revenueCat`，启动时会调用 `configure()`。

建议：

- `NativeBridgeProvider` 首帧不直接 import RevenueCat UI。
- `Purchases` 基础 SDK 可以延后到登录后或会员页前配置。
- `RevenueCatUI` 只在打开 paywall 时动态导入。

收益：

- 减少首包和原生桥初始化压力。
- 避免非会员用户路径加载 paywall UI。

### P1. 语音输入插件按需导入

`lib/native/vn-voice-input.ts` 静态导入：

```ts
@capgo/capacitor-audio-recorder
@capgo/capacitor-speech-recognition
@capacitor/filesystem
```

只要某个设置页或输入组件 import `isNativeSpeechRecognitionAvailable()`，就会把整套语音输入模块拉进对应 chunk。

建议拆分：

- `native-speech-availability.ts`：只负责能力检测，内部动态导入 `SpeechRecognition`。
- `native-audio-recording.ts`：只在录音开始时动态导入 `AudioRecorder`。
- `vn-voice-input.ts` 作为 facade，但不要被设置页直接 import。

收益：

- 设置页只检测能力，不加载录音读文件逻辑。
- 录音能力按用户点击麦克风时加载。

### P1. 本地 warmup model manager 与 local STT model manager 抽公共下载器

`local-ai/warmup-model-manager.ts` 和 `local-stt/local-stt-model-manager.ts` 结构非常相似：

- variant 定义
- manifest 读写
- Filesystem 目录创建
- FileTransfer 下载
- status / install / remove

建议抽一层：

```text
lib/local-models/
  local-model-manager-core.ts
  local-model-types.ts
```

保留业务层：

```text
lib/local-ai/warmup-model-manager.ts
lib/local-stt/local-stt-model-manager.ts
```

收益：

- 降低重复代码。
- 后续新增本地模型能力时，不会复制第三套下载/manifest 逻辑。
- 存储管理页可以统一展示本地模型。

### P1. Profile 设置页拆共同逻辑

`settings-tab.tsx` 和 `mobile-settings-view.tsx` 有重复逻辑：

- 原生语音识别可用性检测。
- 学习提醒刷新。
- 删除账号流程。
- 法律文档加载。
- 偏好开关绑定。

建议抽：

```text
features/profile/hooks/use-native-speech-setting.ts
features/profile/hooks/use-delete-account-flow.ts
features/profile/hooks/use-learning-reminder-setting.ts
features/profile/hooks/use-system-documents.ts
```

收益：

- Web 设置页和移动设置页共享行为，只保留 UI 差异。
- 降低修 bug 时漏改另一端的概率。

### P1. sql-wasm 重复产物检查

`dist` 中同时存在：

- `sql-wasm-1.11.0.wasm`
- `sql-wasm.wasm`

建议确认来源：

- `sql.js`
- `jeep-sqlite`
- `web-sqlite-storage`

如果一个是复制给 wasm loader 的固定文件名，另一个是 Vite asset hash/版本文件，需要确认是否两者都被请求。若只有一个真实使用，另一个应从 build copy 中移除。

## 六、低优先级优化

### P2. 图片资源压缩

`yyzz.png` 约 3.8MB，`logo.png` 约 964KB。建议：

- WebP/AVIF 化。
- 移动端使用独立尺寸。
- 非移动端必需图片从 mobile build 排除。

### P2. 大文件拆分和虚拟列表

源码中超过 1000 行的前端文件不少，尤其后台页面和复杂练习页。后台文件不影响移动端主体验，但移动端热路径里要关注：

- `today-task-page.tsx`
- `learning-plan-page.tsx`
- `learning-unit-page.tsx`
- `member-page.tsx`
- `guided-warmup-phase.tsx`
- `vn-player.tsx`

建议不是“为了行数拆文件”，而是按热路径拆：

- 数据准备 hook
- UI 展示组件
- Drawer/Dialog 子组件
- 纯函数 helpers

这样能减少 re-render 面积，也让 lazy boundary 更容易放置。

### P2. i18n 字典切分

`zh-CN.ts` / `en.ts` / `ja.ts` 都约 1330 行。当前如果启动时加载所有语言，会增加首包和初始化成本。

建议：

- 只同步加载当前语言。
- 其他语言动态导入。
- 系统文档、后台文案可以按路由拆分。

## 七、建议执行顺序

### 第一阶段：包体和首屏

1. 新增 mobile build，排除 admin / portal / company / marketing。
2. 将非首屏用户页继续 lazy。
3. RevenueCat UI 动态导入，只在 paywall 时加载。
4. 检查 ONNX runtime 是否能按需或远程加载。

### 第二阶段：原生能力边界

1. 拆 `vn-voice-input.ts`，让设置页能力检测不加载录音完整实现。
2. 检查所有 Capacitor 插件静态 import，只保留启动硬需求。
3. 继续保持 App pause/resume 的任务暂停策略，不接原生后台下载前不要承诺后台长期执行。

### 第三阶段：代码冗余治理

1. 抽本地模型下载/manifest 公共 core。
2. 抽 Profile 设置页共同 hooks。
3. 拆移动端热路径大页面的业务 hook 和子组件。

### 第四阶段：资源治理

1. 压缩 `logo.png`、`yyzz.png`。
2. 排查两份 `sql-wasm` 是否都需要。
3. 给 `pnpm build` 增加 bundle analyzer 或 `rollup-plugin-visualizer`，每次移动端发布前看包体变化。

## 八、验收指标

建议用这些指标判断优化是否有效：

| 指标 | 当前参考 | 目标 |
|------|---------:|------|
| 移动端 `dist` 总体积 | ~36MB | 先降到 <15MB；若 ONNX 不内置，可进一步下降。 |
| 主入口 JS | ~2.6MB | <1.5MB |
| 首屏可交互 | 需真机测 | iPhone 中端机 <2s，低端 Android <3s |
| 首页进入学习计划 | 需真机测 | 不明显白屏；可接受短 skeleton |
| 学习包下载退后台 | 已暂停可恢复 | 保持当前行为 |
| 音频播放 | 本地优先 + private URL 兜底 | 保持当前行为 |

## 九、不建议现在做的事

- 不建议为了“真后台下载”立刻接原生 Background Transfer。当前暂停/继续逻辑已经符合用户预期，除非学习包体积继续变大。
- 不建议把所有大页面机械拆小。优先拆首包依赖和运行时热路径。
- 不建议默认下载本地 STT 模型。STT 模型太大，必须用户主动选择。
- 不建议让 Remotion / Pixi 后台编辑器能力进入移动端基础包。
