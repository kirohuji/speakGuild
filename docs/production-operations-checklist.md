# 漫语町正式运营上线检查表

这份文档用于判断项目开发完成、数据设计完成后，正式上架 Apple App Store 并开始运营前，还需要补齐哪些能力。原则是：先保证审核能过、线上能稳、用户问题能处理，不做过度设计。

## 当前项目已有基础

从现有代码结构看，项目已经具备不少运营基础：

- 用户端：登录注册、学习首页、学习计划、练习、会员、通知、反馈、账号设置。
- 管理端：用户、会员、账单、通知、反馈、数据统计、内容库、学习包、移动包管理。
- 合规页面：服务条款、隐私政策、儿童隐私、权限说明、SDK 清单、收集信息清单、联系方式。
- 移动端能力：Capacitor iOS 包、推送配置、RevenueCat 订阅、移动包更新、离线学习包。

所以后续重点不是再做一套大而全后台，而是补齐正式运营的最后一公里。

## 必须优先确认

### 1. 账号注销闭环

Apple 要求：如果 App 支持创建账号，就必须允许用户在 App 内发起账号删除。

当前项目已有后端 `/auth/delete-account` 和前端注销入口，但需要重点确认：

- 邮箱/密码注册用户可以正常注销。
- Apple 登录、微信登录等第三方登录用户也可以注销。
- 没有密码的第三方登录用户，不应被“请输入密码”卡住。
- 注销后会删除或匿名化学习记录、收藏、生词本、练习记录、会员关联数据中不再需要保留的部分。
- 法律或财务要求保留的数据，例如订单记录，应有明确保留策略。

建议最小实现：

- 密码用户：输入密码确认注销。
- 第三方登录用户：二次确认后注销，或要求重新完成第三方授权确认身份。
- 注销入口放在：个人中心 -> 设置 -> 账号设置 -> 注销账号。

### 2. iOS 付费只走 Apple IAP / RevenueCat

如果 iOS App 内销售会员、课程、AI 额度、数字内容或订阅，应使用 Apple In-App Purchase。项目已有 RevenueCat 接入，应以它作为 iOS 订阅主链路。

当前上线策略：

- iOS 包内当前只接入 Apple Store 订阅，通过 RevenueCat 完成购买、恢复购买和订阅状态同步。
- 支付宝和微信支付功能先隐藏，不在 iOS 包内展示。
- 支付宝和微信支付代码/能力可以保留，后续用于 Android、Web 或国内渠道包。
- 会员判权统一以后端 `user_membership` 为准，RevenueCat webhook 负责把 Apple 订阅状态同步到后端。

正式上架前需要确认：

- iOS App 内不展示支付宝、微信支付购买入口。
- iOS App 内不引导用户去网页、客服、二维码或外部渠道购买数字内容。
- 会员页提供 Apple IAP 商品。
- 会员页提供“恢复购买”入口。
- 会员页提供“管理订阅”入口。
- RevenueCat webhook 能正确同步订阅、取消、过期、退款、账单问题。
- 前端按平台判断支付入口：iOS Native 只显示 RevenueCat / Apple IAP；Android 或 Web 才允许展示支付宝、微信支付。

支付宝和微信支付可以保留给 Web 或 Android，但 iOS 包内必须按平台隐藏。不要只在文案上隐藏，应确保 iOS Native 用户无法触达外部支付创建订单流程。

### 3. App Store 隐私合规

当前项目已有隐私政策、权限说明和 SDK 清单，但还需要确认 iOS 提交材料一致。

需要准备：

- App Store Connect 隐私标签。
- `PrivacyInfo.xcprivacy` 隐私清单。
- Required Reason API 声明。
- 第三方 SDK 的隐私说明。
- 麦克风、相机、相册、推送、文件存储、语音识别、AI 分析、支付订阅等数据使用说明。

需要特别对齐的功能：

- 账号信息：手机号、邮箱、昵称、头像。
- 学习数据：练习记录、学习进度、收藏、生词本。
- 语音数据：录音、ASR 文本、AI 反馈。
- 设备能力：麦克风、相机、相册、通知。
- 支付数据：RevenueCat / Apple 订阅状态。
- 第三方服务：AI、TTS、STT、云存储、推送、支付 SDK。

注意：隐私政策写了什么、App Store Connect 填了什么、实际代码收集了什么，三者要一致。

### 4. 崩溃和错误监控

正式运营后，最怕用户遇到白屏、支付失败、练习失败，而开发者完全不知道。

建议上线前至少接入一套监控：

- 前端崩溃监控，例如 Sentry。
- 后端异常监控，例如 Sentry 或服务器日志告警。
- Source map 上传，方便定位前端错误。
- API 错误率统计。
- RevenueCat webhook 失败日志。
- AI / TTS / STT 调用失败日志。

最小目标：

- 用户说“打不开”时，能查到是否白屏、接口 500、资源加载失败。
- 用户说“买了会员没到账”时，能查到 RevenueCat 事件和本地会员状态。
- 用户说“练习没有反馈”时，能查到 AI/STT/TTS 调用链路。

### 5. 基础运营告警

不需要复杂大屏，但要有关键异常提醒。

建议至少覆盖：

- 后端服务不可用。
- 数据库连接失败或容量接近上限。
- 文件存储/CDN 访问异常。
- AI 服务失败率异常。
- TTS/STT 服务失败率异常。
- RevenueCat webhook 连续失败。
- 支付/订阅状态同步异常。
- 推送发送失败率异常。

告警方式可以很简单：邮件、飞书、企业微信、Telegram、Server Monitoring 平台都可以。

### 6. 内容发布和回滚流程

项目已有内容库、学习包、移动包管理，所以重点是流程，而不是再造复杂 CMS。

上线前需要明确：

- 谁可以编辑内容。
- 谁可以发布内容。
- 内容发布前如何预览。
- 发现错别字、音频错误、脚本错误时如何修复。
- 发现严重问题时如何下架或回滚。
- 移动包更新失败时如何回滚。

建议最小流程：

1. 内容在管理后台编辑。
2. 发布前用测试账号完整走一遍。
3. 发布后记录版本号、发布时间、发布人。
4. 每个学习包保留上一个可用版本。
5. 线上出问题时优先下架或回滚，不直接在线上热修数据库。

### 7. 客服反馈闭环

项目已有用户反馈页和后台反馈管理，这是很好的起点。

正式运营前需要确认：

- 用户可以提交问题。
- 用户可以看到自己的反馈记录。
- 管理员可以回复。
- 管理员可以修改状态：处理中、已解决、已关闭。
- 严重问题可以被标记或人工升级。
- 用户可以留下联系方式。

建议先不要做复杂客服系统。早期只要做到“收得到、回得去、查得到历史”就够。

### 8. App Review 准备

提交 App Store 前，准备好审核人员能顺利体验的材料。

需要准备：

- 测试账号和密码。
- 如果有会员功能，准备一个可体验高级功能的审核账号。
- App Review Notes 写清楚主要功能路径。
- 如果某些功能依赖网络、AI 服务、订阅沙盒，写清楚测试方式。
- 如果使用麦克风、推送、Apple 登录，确保权限弹窗文案清晰。
- 确保审核环境后端稳定，不要连本地、测试失效域名或临时服务。

## 运营初期建议关注的数据

不要一开始做过度复杂的数据平台。第一阶段只需要看清楚产品是否能用、用户是否留下、钱是否算得清。

建议关注：

- 新增用户数。
- 日活、周活。
- 注册到首次练习转化率。
- 首次练习完成率。
- 次日留存、7 日留存。
- 每日练习完成数。
- 免费用户到会员转化率。
- 订阅购买数、取消数、退款数。
- AI / TTS / STT 人均成本。
- 崩溃率、接口错误率。
- 客服反馈数量和处理时长。

## 可以后置的功能

这些不是一开始正式运营必须做的：

- 复杂 A/B 测试系统。
- 自动化营销系统。
- 用户分群和精细化 CRM。
- 大型 BI 数据仓库。
- 自动内容审核系统。
- 多客服坐席系统。
- 复杂优惠券和活动系统。
- 多级权限审批流。

早期更重要的是产品稳定、内容质量、支付清楚、问题可追踪。

## 漫语町特别注意点

这个项目属于英语学习、AI 口语、会员订阅产品，正式运营时最容易出问题的是三类：

### 审核合规

- 账号注销。
- Apple 登录。
- IAP / RevenueCat。
- 隐私标签。
- 麦克风和语音数据说明。
- 儿童或未成年人相关说明。

### 成本控制

- AI 反馈调用次数。
- STT 语音识别调用次数。
- TTS 音频生成调用次数。
- 失败重试次数。
- 免费用户额度。
- 会员额度和滥用限制。

### 内容质量

- 学习包是否完整。
- 剧本是否能正常跑完。
- 音频是否缺失。
- 图片和背景是否加载正常。
- AI 练习目标是否合理。
- 离线包下载和删除是否稳定。

## 最小上线补全清单

正式运营前，建议按下面清单收口：

- [x] 修第三方登录用户的注销流程。
- [ ] iOS 会员购买只保留 RevenueCat / Apple IAP。
- [ ] iOS 包隐藏支付宝、微信支付入口和外部支付跳转。
- [ ] Android/Web 支付入口与 iOS 支付入口按平台隔离。
- [x] 会员页提供恢复购买入口。
- [x] 会员页提供管理订阅入口。
- [ ] 补齐 `PrivacyInfo.xcprivacy`。
- [ ] 同步 App Store Connect 隐私标签。
- [ ] 接入前端崩溃监控。
- [ ] 接入后端异常监控或日志告警。
- [ ] 为 RevenueCat webhook 增加失败排查记录。
- [ ] 建立后端健康检查。
- [ ] 建立关键服务告警。
- [ ] 明确内容发布和回滚流程。
- [ ] 确认反馈后台可以回复和关闭问题。
- [ ] 准备 App Review 测试账号。
- [ ] 准备 App Review Notes。
