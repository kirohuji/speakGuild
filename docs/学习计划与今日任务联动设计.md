# 学习计划与今日任务联动设计

> 本文记录当前代码中的真实联动方式。结论先说：学习计划负责让用户选择并安装学习包；今日任务离线优先地从已安装学习包中抽题。二者主要通过本地缓存和学习包 `packId` 联动，而不是通过一张独立的“每日计划表”强绑定。
>
> 最后更新：2026-07-01

## 一、核心概念

| 概念 | 当前代码中的含义 | 主要来源 |
| --- | --- | --- |
| 学习单元 / 学习包 | 一个 `Scene`，包含 `TrainingTopic`、词汇、句块、句型、剧本入口等内容。 | 后端 `LearningService.getLearningUnits()` / `getLearningUnitDetail()` |
| 我的学习 / 学习计划 | 用户已经开始学习的 `Scene` 列表。后端以 `UserSceneProgress` 表示“已加入计划”。 | `/learning/my-units` |
| 今日任务 | 当天从某个已安装学习包的 `trainingTopics.metadata.outputTraining.pipeline` 中抽出的输出练习列表。 | 前端 `TodayTaskPage` |
| 学习包安装状态 | 客户端本地是否已安装 zip / manifest 内容。 | SQLite `downloaded_packs` |
| 今日完成状态 | 当天已完成的题目、attempt 和 run 状态。 | SQLite `daily_practice_items` / `daily_practice_attempts` / `daily_practice_runs` |
| 今日练习记录 | 今日任务和知识点练习共用的 warmup 作答明细、最近答案、同步状态。 | SQLite `warmup_records` / `warmup_record_entries` |

## 二、产品层设计

学习计划和今日任务的关系是：

1. 用户在学习计划页选择一个学习包。
2. 前端调用开始学习接口，后端创建或复用 `UserSceneProgress`。
3. 前端把学习包加入下载队列，安装到本地 SQLite / 资产缓存。
4. 今日任务页读取本地已安装学习包。
5. 今日任务页从学习包的训练配置中生成当天练习。
6. 用户完成今日任务后，本地记录进度和练习结果，并尝试同步后端。

所以当前设计更像：

```text
学习计划 = 用户选择的内容池
今日任务 = 从当前内容池里按天抽取的输出训练
```

这不是一个服务端每日调度系统。当前 `TodayTaskPage` 不调用 `/learning/today`，而是直接从离线学习包生成页面内容。

## 三、后端链路

### 1. 学习计划列表

入口：

- `GET /learning/my-units`
- 控制器：`apps/backend/src/modules/learning/learning.controller.ts`
- 服务：`apps/backend/src/modules/learning/learning.service.ts`

后端查询当前用户的 `UserSceneProgress`，并带出关联 `Scene`、分类、`TrainingTopic` 和数量统计。也就是说，只要某个 `Scene` 有了用户进度记录，它就会出现在学习计划里。

进度字段包括：

- `vocabLearned`
- `chunkMastered`
- `completedPracticeCount`
- `completedScriptCount`
- `readiness`
- `mastery`
- `completionPercent`

其中 `completedPracticeCount` 当前主要从已分析且分数超过阈值的 `PracticeSession` 推导。

### 2. 开始学习

入口：

- `POST /learning/units/:id/start`

行为：

- 校验学习包访问权限。
- 最多允许同时学习 3 个未完成单元。
- 对 `UserSceneProgress` 做 upsert。
- 已经存在进度的学习包可以继续学习。

这一步是“加入学习计划”的服务端依据。

### 3. 学习包详情和离线包

入口：

- `GET /learning/units/:id`
- `GET /learning/units/:id/offline-manifest`
- `GET /learning/units/:id/download-pack`
- `GET /learning/packs/check`
- `GET /learning/units/:id/download-delta`

今日任务需要的关键字段来自学习包详情中的：

```text
UnitDetail.trainingTopics[].metadata.outputTraining
```

其中 `outputTraining.pipeline` 是今日任务真正抽题的数据源。

### 4. 服务端今日任务接口

后端仍有：

- `GET /learning/today`
- `LearningService.getTodayTasks()`

它会选择最近更新且未完成的 `UserSceneProgress`，没有进行中进度时回退到第一个可解锁 `Scene`，然后生成词汇、句块、句型、开口练习、剧本挑战等任务。

但当前前端 `TodayTaskPage` 没有调用 `learningApi.getTodayTasks()`。因此这条接口目前更像旧版或备用设计，不能代表当前页面的主链路。

## 四、前端学习计划链路

主要文件：

- `apps/frontend/src/features/learning/pages/learning-plan-page.tsx`
- `apps/frontend/src/features/learning/components/my-learning-view.tsx`
- `apps/frontend/src/stores/learning.store.ts`
- `apps/frontend/src/lib/offline/learning.repository.ts`
- `apps/frontend/src/lib/offline/learning-pack.service.ts`

页面加载时：

1. `LearningPlanPage` 调用 `fetchMyLearning()` 和 `fetchDownloadedPacks()`。
2. `fetchMyLearning()` 先读 `learningRepository.getMyUnits()`。
3. `learningRepository.getMyUnits()` 优先返回本地 `my_learning_units`，本地没有时请求 `/learning/my-units`。
4. `MyLearningView` 根据 `downloaded_packs` 判断学习包是否已安装。
5. 未安装时，学习单元卡片会显示下载动作，并阻止进入学习单元详情。

用户加入学习包时：

1. `enrollUnit(unitId)` 调用 `learningRepository.enrollUnit()`。
2. repository 先把摘要写入本地 `my_learning_units`，并把 `my_unit:create` 放入 outbox。
3. 在线时调用 `/learning/units/:id/start`，成功后刷新 `/learning/my-units`。
4. store 将学习包加入下载队列。
5. 下载队列调用 `learningPackService.installUnit()`。
6. 安装完成后刷新 `downloaded_packs` 和 `myUnits`。

这保证学习计划页和今日任务页共享同一批本地学习包数据。

## 五、今日任务生成链路

主要文件：

- `apps/frontend/src/features/learning/pages/today-task-page.tsx`
- `apps/frontend/src/lib/offline/learning.repository.ts`
- `apps/frontend/src/lib/offline/practice.repository.ts`
- `apps/frontend/src/stores/warmup-session.store.ts`

### 1. 选择哪个学习包

`TodayTaskPage` 的选择顺序：

1. 如果 URL 有 `packId`，优先读取该学习包的本地详情。
2. 否则读取本地 `downloaded_packs` 和 `my_learning_units`。
3. 优先选择“我的学习中第一个已安装包”。
4. 如果没有匹配的我的学习包，则选择第一个已安装包。
5. 如果没有任何已安装包，显示空态，引导去学习计划选择学习包。

当前路由支持：

```text
/today
/today?packId=<sceneId>
```

不过现有学习计划卡片主要跳转到 `/learning/units/:id`，没有在学习计划页直接跳 `/today?packId=...`。

### 2. 从哪里抽题

今日任务遍历：

```text
unit.trainingTopics[].metadata.outputTraining.pipeline[]
```

支持的题型来自 `TYPE_META`：

- `chunk_substitution`：句块替换 / 词汇替换
- `vocab_drill`：词汇输出
- `vocab_sentence_building`：一词多句
- `pattern_drill`：句型操练
- `sentence_decomposition`：句子拆解

每个 pipeline item 会随机抽一个可练习条目。`runSeed` 默认来自当天日期的 hash，所以同一天初次进入会尽量稳定；点击换一批会改成随机种子重新生成。

### 3. 每日题量

今日任务读取用户偏好：

```text
usePreferencesStore((s) => s.dailyGoal)
```

生成策略：

1. 先遍历学习包里所有训练话题和 pipeline。
2. 按题型保证多样性，每种已有题型尽量先取 1 道。
3. 再随机补足到 `dailyGoal`。
4. 如果某些题型完全没有配置，会生成占位卡片，提示后台补题；占位卡片可以跳过并标记完成。

### 4. 今日进度与练习记录恢复

当前实现把今日任务拆成两类本地状态：

| 状态 | 本地表 | 作用 |
| --- | --- | --- |
| 今日任务进度 | `daily_practice_items` / `daily_practice_attempts` / `daily_practice_runs` | 记录每个知识点练习 item 的复习排期、attempt、当天 run 和完成状态。 |
| 每题作答明细 | `warmup_records` | 记录用户答案、反馈、纠正、录音、分数等，用于今日练习记录 drawer 和返回上一题恢复。 |

页面再次打开时，不直接依赖 React 内存状态，而是：

1. `TodayTaskPage` 根据 `date + mode + pack scope + scheduledItemIds` 生成本次今日任务的 `warmupRecordId`。
2. 从 SQLite `warmup_records` 读取对应记录。
3. 调用 `warmupStore.hydrateSession(records)`。
4. 恢复今日练习记录 drawer，以及每道题的输入、反馈、通过/失败状态。

知识点热身页使用类似机制，但 `warmupRecordId` 以 `topicId` 为主：

```text
guided-warmup:<topicId>:records
```

### 5. 前端到后端的同步流程

```mermaid
flowchart TD
  A[前端页面<br/>今日任务 / 知识点热身] --> B[用户提交某一道题]
  B --> C[AI 判题返回反馈]
  C --> D[warmupStore.recordStep / recordEntry<br/>内存保存本题状态]

  D --> E[SQLite: warmup_records<br/>保存每题作答明细<br/>syncStatus = pending]
  D --> F[SQLite: daily_practice_items / attempts / runs<br/>保存复习进度、完成状态、attempt]

  E --> G[页面重新打开]
  G --> H[从 SQLite warmup_records 读取]
  H --> I[warmupStore.hydrateSession]
  I --> J[恢复 drawer 记录 + 每题输入/反馈/通过状态]

  E --> K[offlineSyncService.flush]
  F --> K

  K --> L[syncOutbox pending 队列]
  L --> M[/sync/push]

  M --> N[后端 PracticeWarmupRecord<br/>保存每题作答明细]
  M --> O[后端 UserDailyPracticeRun<br/>保存今日任务 run]
  M --> P[后端 UserDailyPracticeAttempt<br/>保存每题 attempt]
  M --> Q[后端 UserWarmupItemProgress<br/>保存复习排期/掌握度]

  N --> R[后端返回 synced]
  O --> R
  P --> R
  Q --> R

  R --> S[前端本地标记同步完成<br/>warmup_records syncStatus = synced<br/>attempt syncStatus = synced]

  T[之后启动/手动同步] --> U[offlineSyncService.pull]
  U --> V[/sync/pull?cursor=...]
  V --> W[后端返回增量数据<br/>practiceWarmupRecords + progress/session 等]
  W --> X[前端写回 SQLite]
  X --> H
```

按职责分层：

```mermaid
flowchart LR
  A[React 页面] --> B[warmupStore<br/>当前会话状态]
  B --> C[SQLite<br/>warmup_records]
  B --> D[SQLite<br/>daily_practice_*]

  C --> E[syncOutbox / offlineSync]
  D --> E

  E --> F[/sync/push]
  F --> G[后端数据库]

  G --> H[/sync/pull]
  H --> C
  H --> D
```

### 6. 更新规则

用户每提交一题：

1. 更新 `warmupStore`，让当前页面立即显示反馈。
2. 写入 SQLite `warmup_records`，使 drawer 和返回上一题能恢复。
3. 更新 SQLite `daily_practice_items`、`daily_practice_attempts`、`daily_practice_runs`，用于今日进度和复习排期。

页面重新进入：

1. 根据当前任务生成稳定 `warmupRecordId`。
2. 读取 SQLite `warmup_records`。
3. 调用 `hydrateSession()` 回灌 `warmupStore`。

同步到后端：

1. `offlineSyncService.flush()` 读取 `syncOutbox`。
2. `warmup_records:create` 通过 `POST /sync/push` 写入后端 `PracticeWarmupRecord`。
3. `daily_practice` 通过 `POST /practice/daily-practice/complete` 写入后端 `UserDailyPracticeRun`、`UserDailyPracticeAttempt`、`UserWarmupItemProgress`，并可带一份 `warmupRecord`。
4. 同步成功后，本地 `warmup_records.syncStatus` 和 `daily_practice_attempts.syncStatus` 标记为 `synced`。

从后端拉取：

1. `offlineSyncService.pull()` 调用 `GET /sync/pull?cursor=...`。
2. 后端返回 `practiceWarmupRecords` 等增量数据。
3. 前端写回 SQLite `warmup_records`。
4. 页面之后仍从 SQLite 恢复，不直接依赖网络返回。

## 六、离线缓存关系

当前联动依赖这些本地表：

| 表 | 作用 |
| --- | --- |
| `my_learning_units` | 学习计划页和今日任务页都用来判断“我的学习”。 |
| `downloaded_packs` | 判断学习包是否安装，今日任务只消费已安装包。 |
| `downloaded_unit_details` | 保存学习包详情和 topic 详情，是今日任务抽题的数据源。 |
| `warmup_records` | 保存今日任务/知识点热身每题作答明细、反馈、纠正和同步状态。 |
| `daily_practice_items` | 保存每个知识点练习 item 的复习排期和掌握度。 |
| `daily_practice_attempts` | 保存每次练习 attempt，用于后端同步和统计。 |
| `daily_practice_runs` | 保存当天生成的任务集合、完成集合和 run 统计。 |
| `daily_activity` | 保存当天练习活跃次数，用于打卡统计。 |
| `sync_outbox` | 离线或同步失败时暂存创建/删除/练习记录等操作。 |

## 七、内容配置要求

如果一个学习包希望能出现在今日任务中，仅有 `TrainingTopic` 不够，还需要在 topic metadata 中配置输出训练：

```json
{
  "outputTraining": {
    "enabled": true,
    "pipeline": [
      {
        "id": "example-step",
        "type": "chunk_substitution",
        "chunk": "take care of",
        "chunkMeaning": "照顾",
        "items": [
          {
            "zh": "我需要照顾我的室友。",
            "answer": "I need to take care of my roommate."
          }
        ]
      }
    ]
  }
}
```

缺少 pipeline 的学习包仍然可以加入学习计划和下载，但今日任务会没有真实题目，或只显示占位题型。

## 八、当前边界和注意事项

1. `TodayTaskPage` 当前不使用 `/learning/today`，因此后端今日任务的词汇/句块限额逻辑不会影响当前页面。
2. 今日任务要求学习包已安装。本地没有 `downloaded_packs` 和 `downloaded_unit_details` 时，页面无法生成任务。
3. 学习计划允许最多 3 个未完成单元；今日任务默认只选其中一个包，不会自动混合多个学习包。
4. `packId` 可以强制指定今日任务来源，但学习计划页当前没有显式入口跳转到 `/today?packId=...`。
5. 同一天默认题目相对稳定；“换一批题目”只更换当前今日任务 run 的题目集合，不清空此前今日练习记录。若知识点练习完成了不在当前 20 题内的题目，今日记录会显示额外完成数，但待练集合仍以当前 run 为准。
6. 今日任务完成后保存的是 warmup 记录；它不会直接调用 `/learning/units/:id/progress` 更新 `UserSceneProgress`。
7. 服务端 `UserSceneProgress.completedPracticeCount` 主要基于正式 practice session 的分析结果推导，和今日任务 warmup 完成记录不是同一个进度来源。

## 九、后续可选优化

如果要让设计更一致，可以考虑：

1. 明确废弃或改造 `/learning/today`，避免服务端和前端存在两套今日任务模型。
2. 在学习计划卡片增加“今日练这个包”入口，跳转 `/today?packId=<unit.id>`。
3. 今日任务完成后同步影响 `UserSceneProgress`，或者在学习计划页单独展示 warmup 完成进度。
4. 将今日任务的选包策略抽到 repository 或 store，避免页面组件承担业务决策。
5. 后台学习包编辑页增加 `outputTraining.pipeline` 完整性检查，减少占位题型。

## 十、下一阶段设计决策

本节根据 2026-06-21 的产品讨论补充，作为后续实现依据。

### 1. 废弃 `/learning/today`

决策：废弃服务端 `/learning/today` 今日任务模型。

原因：

- 当前前端 `TodayTaskPage` 已经不调用 `/learning/today`。
- 服务端 `/learning/today` 生成的是词汇、句块、句型、practice、script 混合任务；前端实际生成的是 `outputTraining.pipeline` 的知识点输出练习。
- 两套模型会导致“今日任务到底是什么”无法解释，也会让后续学习算法和统计口径分裂。

推荐处理：

1. 前端移除 `learningApi.getTodayTasks()`、`TodayPlan`、`TodayTask` 等未使用类型。
2. 后端删除 `LearningController.getTodayTasks()` 和 `LearningService.getTodayTasks()`。
3. 如果担心旧客户端仍调用，可以短期保留接口但返回 `410 Gone` 或明确错误：`/learning/today has been deprecated; use offline daily task planner`。
4. 新的后端同步接口不要叫 `/learning/today`，避免语义混淆；建议使用 `/learning/daily-progress` 或 `/practice/warmup-progress`。

### 2. 不增加“今日练这个包”按钮

决策：学习计划页不增加“今日练这个包”按钮。

原因：

- 用户已经有底部导航的“今日任务”入口。
- 学习计划页的核心职责是管理学习包：开始、下载、更新、退出。
- 如果在每个学习包卡片加“今日练这个包”，会暗示用户每天要手动选择包，而当前目标更像“系统根据学习算法自动安排今日内容”。

保留能力：

- `/today?packId=<unit.id>` 可以作为调试或深链能力保留，但不作为主 UI 暴露。
- 今日任务默认选包策略应从页面组件抽出，后续交给学习算法决定，而不是靠按钮临时切换。

### 3. 今日任务和学习计划进度的关系

决策：今日任务完成后需要同步到后端，但不能混入 `PracticeSession` 口径。

需要区分三类练习：

| 类型 | 当前表 | 含义 | 是否影响学习计划话题进度 |
| --- | --- | --- | --- |
| 知识点练习 / warmup | `PracticeWarmupRecord` | 今日任务里的句块、词汇、句型、拆句等轻量输出练习。 | 应该影响“知识点练习进度”，但不应算作 VN 对话练习。 |
| VN / 正式话题练习 | `PracticeSession` | 进入 `/practice/session/:topicId` 的对话式练习和 AI 分析。 | 影响正式话题练习完成度。 |
| 剧本练习 | `StoryRecord` | VN 剧情 / script play 记录。 | 影响剧本完成度。 |

当前代码中，`UserSceneProgress.completedPracticeCount` 主要由 `PracticeSession` 分析结果推导。今日任务的 warmup 完成记录不会直接更新这个字段。

下一步建议新增独立统计口径：

- `warmupCompletedCount`：该学习包下已完成知识点练习数。
- `warmupDueCount`：今天应该复习/练习的知识点数。
- `warmupOverdueCount`：以前应该完成但未完成、累计到今天的知识点数。
- `warmupMastery`：知识点练习掌握度，可由题目通过率、最近复习状态、连续完成情况计算。

这些字段可以作为后端接口返回的统计结果，不一定全部落在 `UserSceneProgress` 上。`UserSceneProgress` 更适合继续保留宏观学习包进度。

### 4. 今日任务底部话题卡片进度条

决策：在 `/today` 底部“当前学习话题”卡片增加轻量进度条和状态色，而不是回到学习计划页增加入口按钮。

建议 UI：

- 每个话题卡片底部增加一条细进度条。
- 进度条前半或主体显示该话题知识点练习完成度。
- 卡片右侧或标题下方显示状态 badge。
- 不在卡片里塞太多文字，避免今日任务页变成学习计划页。

建议状态色：

| 状态 | 颜色 | 含义 |
| --- | --- | --- |
| 今日待练 | 蓝色 | 今天首次安排的新练习。 |
| 今日待复习 | 琥珀色 | 到了复习时间，需要巩固。 |
| 已逾期累积 | 红色 | 昨天或更早应完成但未完成，今天继续累积。 |
| 今日已完成 | 绿色 | 今天已完成该话题的今日任务。 |
| 长期掌握 | 紫色或中性灰 | 最近多次通过，短期无需复习。 |

话题卡片可展示：

- `todayNewCount`：今日新练数量。
- `todayReviewCount`：今日复习数量。
- `overdueCount`：逾期累计数量。
- `doneTodayCount` / `scheduledTodayCount`：今日完成比例。
- `topicWarmupProgress`：该 topic 的知识点练习整体完成度。

颜色优先级建议：

```text
overdueCount > 0       -> 红色
todayReviewCount > 0   -> 琥珀色
todayNewCount > 0      -> 蓝色
doneTodayCount 完成全部 -> 绿色
短期无需复习           -> 中性或紫色
```

### 5. 今日任务学习算法

目标：今日任务不只是随机抽题，而是能安排新练、复习、逾期累积，并且离线可用。

当前实现把今日任务拆成两层：

1. 候选池：`dailyPracticeRepository.resolveCandidateUnits()` 根据用户设置和可选 `packId` 决定从哪个已安装学习包取内容，再从 `outputTraining.pipeline` 展开成可练习 item。
2. 排程层：`dailyPracticeRepository.buildTodayPlan()` 根据每个 item 的学习状态决定今天练什么，并写入 `daily_practice_runs`。

`TodayTaskPage` 不直接决定选哪个学习包，也不直接写排程算法；页面只消费 `useDailyPracticeStore.plan`。

每个可练习 item 使用稳定 ID：

```text
packId:topicId:itemKey:type[:patternPart]:i-promptKey
```

生成规则：

- 优先使用真实 `item.id`、`prompt.id`、`prompt.vocabId`。
- 没有真实 ID 时，对题目核心内容做稳定 hash。
- 前端 `createWarmupPracticeItemId()` 和后端 `LearningService` 中的 warmup itemId 生成逻辑保持一致。
- 本地和后端都围绕这个 ID 记录状态，避免题目每天重排、随机顺序变化或跨入口练习后无法对齐。

状态字段：

| 字段 | 含义 |
| --- | --- |
| `itemId` | 稳定题目 ID。 |
| `packId` / `topicId` | 所属学习包和话题。 |
| `type` | 题型，如 `chunk_substitution`、`vocab_drill`。 |
| `status` | `new`、`learning`、`review`、`mastered`。 |
| `dueDate` | 下次应练日期。 |
| `lastPracticedAt` | 最近练习时间。 |
| `streak` | 连续通过次数。 |
| `lapseCount` | 失败或遗忘次数。 |
| `easeFactor` | 间隔重复难度系数。 |
| `intervalDays` | 当前复习间隔。 |
| `lastPracticedAt` 是否为今天 | 今天是否完成。 |
| `lastScore` | 最近一次表现，可用 `strong`、`ok`、`weak` 或 0-100。 |

排程策略：

1. `review` 模式：取所有 `overdue + review`，复习无上限。
2. `practice` 模式：从非 mastered 且今天未 done 的池子里取题，数量受 `dailyGoal` 限制。
3. 逾期项不会消失，会继续累积到后续日期。
4. 用户可以选择随机或顺序；随机只影响当前 run 的题目集合，不清空今日已有练习记录。
5. 同一天、同 mode、同 scope、同 packIds 时优先复用 `daily_practice_runs` 中的 `scheduledItemIds`，避免刷新页面就换题。
6. 用户完成后根据表现更新 `dueDate`：
   - `strong`：间隔扩大，首次通常到 3 天，之后按更大倍率增长，最高 30 天。
   - `ok`：小幅扩大或维持，最高 14 天。
   - `weak` / `miss`：明天继续复习，重置 streak 或增加 lapse。
7. 如果今天没完成，未完成项保留原 `dueDate`，明天继续出现在 due/overdue 池中。

完成后的更新链路：

```mermaid
flowchart TD
  A[完成某题] --> B[dailyPracticeRepository.completeItem / completeAdHocItem]
  B --> C[SQLite daily_practice_items<br/>更新 dueDate / status / streak]
  B --> D[SQLite daily_practice_attempts<br/>记录 attempt]
  B --> E[SQLite daily_practice_runs<br/>如果题目属于当前 run，则加入 completedItemIds]
  C --> F[/practice/daily-practice/complete]
  D --> F
  E --> F
  F --> G[后端 UserWarmupItemProgress / Attempt / Run]
  G --> H[重算 UserSceneProgress.completedPracticeCount]
```

`UserSceneProgress.completedPracticeCount` 是幂等重算，不是简单累加：后端按 `packId + bestScoreRank >= 2` 统计已完成 warmup item 数，再写回学习计划进度，避免重复同步导致计数膨胀。

### 6. 离线优先和后端同步

决策：今日任务学习状态必须本地先写、后端后同步。

本地建议新增或改造：

| 本地表 | 作用 |
| --- | --- |
| `daily_practice_items` | 每个稳定题目的复习状态、dueDate、streak、interval 等。 |
| `daily_practice_runs` | 每天生成的一次任务计划，记录 pack/topic、itemIds、状态。 |
| `daily_practice_attempts` | 每次作答记录，支持离线回放同步。 |

后端建议新增镜像表或等价模型：

| 后端模型 | 作用 |
| --- | --- |
| `UserWarmupItemProgress` | 用户对每个 warmup item 的长期状态。 |
| `UserDailyPracticeRun` | 某天生成的计划快照，便于后台统计和跨设备恢复。 |
| `UserDailyPracticeAttempt` | 每次作答记录，便于统计、评估、排查。 |

同步规则：

1. 练习完成时先写本地 `daily_practice_attempts` 和 item progress。
2. 同步任务写入 `sync_outbox`。
3. 在线时批量 upsert 到后端。
4. 后端使用 `clientAttemptId` 或 `runId + itemId + attemptIndex` 去重，保证重复同步不会重复计数。
5. 后端返回合并后的 item progress，前端再更新本地。
6. 离线多天时，未完成 due item 不丢失，按本地 dueDate 继续累积。

当前 `PracticeWarmupRecord` 可以继续保存一次练习的汇总和 AI 反馈，但不建议单独承担学习算法状态。它更像历史记录，不是调度状态。

注意：`PracticeWarmupRecord` 现在应通过统一同步链路写入后端，即本地 `warmup_records` → `sync_outbox` → `POST /sync/push` → 后端 `PracticeWarmupRecord`。旧的 `submitWarmupRecords()` / `warmupRecordApi.save()` / `warmupRecordApi.assess()` 只能作为兼容入口，不应成为新页面的主同步路径。

### 7. 后台用户详情 Tab 设计

决策：后台用户详情 dialog 可以改为多 tab，用于管理和观察用户学习情况。

当前用户详情只展示基础资料、登录会话、账号、学习画像、练习/VN总数和 AI 用量。后续建议拆成：

| Tab | 内容 |
| --- | --- |
| 概览 | 头像、角色、会员、在线状态、输出等级、XP、积分。 |
| 学习包 | 用户加入的学习包、安装/同步状态、总体进度、最近练习时间。 |
| 知识点练习 | warmup 完成数、复习 due/overdue、平均分、最近记录、按学习包/话题聚合。 |
| VN 练习 | `PracticeSession` 数量、通过话题、平均分、最近分析结果。 |
| 剧本练习 | `StoryRecord` 完成、通过、回合数、XP。 |

后端不要把所有学习统计塞进 `GET /admin/users/:id`，否则用户详情 dialog 首次打开会很重。当前采用聚合摘要接口：

```

返回学习包、warmup、VN、剧本的摘要。后续如果某个 tab 需要明细，再按 tab 懒加载拆接口：

GET /admin/users/:id/warmup-stats
GET /admin/users/:id/practice-stats
GET /admin/users/:id/story-stats
GET /admin/users/:id/ai-usage
```

也就是说：详情首屏用 `learning-overview`，明细列表按需加载。

## 十一、歧义和冲突检查

目前需求整体不冲突，但有几处需要实现前明确：

1. “废弃 `/learning/today`”和“今日任务需要同步到后端”不冲突。废弃的是服务端生成今日任务的旧模型；新增的是同步今日任务状态和学习算法状态的接口。
2. “不需要今日练这个包按钮”和“今日任务底部话题进度条”不冲突。前者避免学习计划页增加入口；后者是在今日任务页解释当前安排和进度。
3. “不要和 practice session 混淆”和“学习计划要显示知识点练习完成度”不冲突。应新增 warmup/知识点进度口径，而不是复用 `PracticeSession.completedPracticeCount`。
4. “今日没完成明天继续累积”要求学习算法必须有稳定 item ID 和 dueDate，不能继续只靠当天随机抽题。
5. “离线也能流程”要求调度算法和状态更新首先在前端本地完成，后端只能作为同步镜像和跨设备恢复来源。

已确认的产品细节：

1. 今日任务排程范围做成用户自己的移动端设置：默认只从一个学习包排程；开启“跨学习包混合排程”后，允许从多个已安装学习包混合安排。
2. 复习无上限，新题受 `dailyGoal` 限制。也就是说逾期和到期复习不会因为目标数而消失，新题只在复习压力较低时补充。
3. warmup 通过标准直接使用题目组件返回的 `strong`、`ok`、`weak`。
4. 学习计划页总体 `completionPercent` 暂不混入 warmup 掌握度，避免把“知识点复习”和“学习包宏观完成度”绑死。warmup 进度先在 `/today` 话题卡片和后台用户详情里独立展示。
5. 跨设备合并时，后端保留所有 attempt 记录；长期 item progress 按“最高分 + 最新练习时间”合并。

## 十三、最新闭环设计（2026-07-01）

本节是当前实现的最新口径：今日任务 dialog 和知识点练习 dialog 已统一，练习记录按同一套 warmup 数据模型保存、去重、覆盖更新和同步。

### 1. 统一题目身份

同一道题的身份以稳定 `stepId/itemId` 为准，而不是页面顺序。

```text
stepId/itemId = createWarmupPracticeItemId(...)
优先使用真实 item.id
没有真实 id 时使用题目核心内容 hash
```

这样即使今日任务随机打乱顺序，或知识点练习按 topic 顺序展示，只要题目内容相同，就会落到同一个 `stepId`。重练当前题不会新增一条重复题目，而是覆盖更新同一题的最新答案，同时递增 `practiceCount`。

### 2. 当前 session 与历史答案读取规则

页面展示一题时按以下顺序恢复：

```text
1. 当前 warmup session 的 stepStates / records
2. 当前 session 对应的 warmup_records
3. 如果当前 session 没有，再按 stepId/itemId 查最近一次历史作答
4. 找到后显示为“已练过 / 最近答案”，并允许重新练习覆盖更新
```

这保证了：

- 今日任务练过的题，在知识点练习里会显示已练过和最近答案。
- 知识点练习练过的题，在今日任务里遇到同一题也能恢复最近答案。
- 当前 session 的作答优先，避免用户本轮刚改过答案却被历史记录覆盖。

### 3. 今日任务与知识点练习的联动

```mermaid
flowchart TD
  A[今日任务 Dialog] --> C[warmupStore.recordStep / recordEntry]
  B[知识点练习 Dialog] --> C

  C --> D[SQLite warmup_records<br/>按 recordId 保存整组记录]
  C --> E[SQLite warmup_record_entries<br/>按 stepId 拆分索引]
  C --> F[SQLite daily_practice_items / attempts / runs<br/>仅今日任务排程状态]

  E --> G[今日练习记录 Drawer<br/>按日期读取 + stepId 去重]
  E --> H[知识点练习记录 Tab<br/>按历史记录读取 + 题目标题展示]
  E --> I[题目打开时<br/>按 stepId 查最近答案]

  D --> J[syncOutbox / offlineSync]
  F --> J
  J --> K[/sync/push 或 daily-practice complete]
  K --> L[后端 PracticeWarmupRecord / UserDailyPractice*]
  L --> M[/sync/pull]
  M --> D
  M --> E
```

今日任务和知识点练习共享 `warmup_records`，但 `daily_practice_*` 只负责今日任务的排程状态。因此如果用户在知识点练习里完成了一个不属于当前今日随机 20 题的题目：

- 今日练习记录会出现这条记录，并体现在“额外完成”统计里。
- 当前今日任务的待练列表不变，因为它不在当前 run 的 `scheduledItemIds` 中。
- 之后如果今日任务随机抽到同一题，会直接显示最近答案，并可重新练习覆盖更新。

### 4. 表结构与职责

| 表 | 职责 | 关键查询 |
| --- | --- | --- |
| `warmup_records` | 一次 warmup session 的整组作答记录，包含 items、topic、syncStatus、remoteId。 | 按 `id` 恢复当前 session；按 topic/时间列历史记录。 |
| `warmup_record_entries` | 从 `warmup_records.items` 拆出的题目级索引表。 | 按 `practiced_date` 查今日记录；按 `step_id` 查最近答案；按 `record_id` 覆盖删除。 |
| `daily_practice_items` | 每个稳定题目的长期复习状态。 | 按 `item_id` 查 dueDate、status、streak、interval。 |
| `daily_practice_attempts` | 今日任务每次作答 attempt。 | 同步后端统计和去重。 |
| `daily_practice_runs` | 某天某次今日任务的计划快照。 | 保存当前随机/顺序生成的 `scheduledItemIds` 和完成集合。 |
| `daily_activity` | 打卡活跃计数。 | 今日练习活跃统计。 |

`warmup_record_entries` 是移动端性能优化表，不替代 `warmup_records`。前者让高频读取不再扫描大 JSON，后者保留完整 session 结构和同步 payload。

### 5. 覆盖更新与去重规则

```text
同 stepId 多次重练:
  更新 warmupStore 当前题状态
  覆盖 warmup_records.items 中同 stepId 的条目
  删除 warmup_record_entries 中同 recordId 的旧 entries
  重新写入 entries
  practiceCount + 1

今日记录 Drawer:
  按 practiced_date 读取 entries
  按 stepId 去重
  保留 recordUpdatedAt 最新的一条
  显示题型 + 来源（今日练习 / 知识点练习）
```

### 6. 自动下一题

自动下一题必须等待：

```text
用户提交
  -> AI 判题成功返回
  -> recordStep / recordEntry 写入成功
  -> 当前题 UI 显示反馈
  -> 才进入下一题
```

失败、网络异常或本地写入异常时不自动跳题，避免用户以为题目已完成但记录丢失。

### 7. 性能设计

移动端热路径不再扫描 `warmup_records`：

| 场景 | 旧设计 | 最新设计 |
| --- | --- | --- |
| 打开今日练习记录 | `list(warmup_records)` 后展开所有 items | `findByIndex('warmup_record_entries', 'practiced_date', today)` |
| 打开题目恢复最近答案 | 扫所有 warmup_records 并按 stepId 过滤 | `findByIndexIn('warmup_record_entries', 'step_id', stepIds)` |
| 重练覆盖更新 | 全表 hydrate 后过滤删除 entries | `DELETE FROM warmup_record_entries WHERE record_id = ?` |

这些风险目前已经处理到可控范围：

1. `warmup_records.items` 不再作为今日记录和最近答案的查询入口；它只保留当前 session 的完整 payload。今日任务默认 20 题，知识点练习也按当前组写入，单条 JSON 不会无限承担全局历史查询压力。
2. `warmup_record_entries` 是题目级索引表。重练同一题是覆盖更新同一 `recordId + stepId` 的 entry，并递增 `practiceCount`，不会因为“重新练习当前题”制造重复展示项。
3. 今日记录 drawer 按日期索引查，题目恢复按 stepId 批量 `IN` 查，UI 再按 stepId 去重。历史增长不会让打开 drawer 退回全表扫描。
4. 剩余可选维护项是长期本地历史清理，例如只保留最近 N 天或最近 N 条 warmup 历史；这属于存储空间治理，不是当前热路径性能问题。
5. 同步时网络慢不会影响当前页面，因本地先写；后端同步失败会留在 outbox，存储管理里可清理测试数据。
