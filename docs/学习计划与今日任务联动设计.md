# 学习计划与今日任务联动设计

> 本文记录当前代码中的真实联动方式。结论先说：学习计划负责让用户选择并安装学习包；今日任务离线优先地从已安装学习包中抽题。二者主要通过本地缓存和学习包 `packId` 联动，而不是通过一张独立的“每日计划表”强绑定。

## 一、核心概念

| 概念 | 当前代码中的含义 | 主要来源 |
| --- | --- | --- |
| 学习单元 / 学习包 | 一个 `Scene`，包含 `TrainingTopic`、词汇、句块、句型、剧本入口等内容。 | 后端 `LearningService.getLearningUnits()` / `getLearningUnitDetail()` |
| 我的学习 / 学习计划 | 用户已经开始学习的 `Scene` 列表。后端以 `UserSceneProgress` 表示“已加入计划”。 | `/learning/my-units` |
| 今日任务 | 当天从某个已安装学习包的 `trainingTopics.metadata.outputTraining.pipeline` 中抽出的输出练习列表。 | 前端 `TodayTaskPage` |
| 学习包安装状态 | 客户端本地是否已安装 zip / manifest 内容。 | SQLite `downloaded_packs` |
| 今日完成状态 | 当天已完成的题目 ID 集合。 | SQLite `daily_progress` |
| 今日练习记录 | 今日任务完成后保存的 warmup 练习明细和同步状态。 | SQLite `warmup_records` |

## 二、产品层设计

学习计划和今日任务的关系是：

1. 用户在学习计划页选择一个学习包。
2. 前端调用开始学习接口，后端创建或复用 `UserSceneProgress`。
3. 前端把学习包加入下载队列，安装到本地 SQLite / 资产缓存。
4. 今日任务页读取本地已安装学习包。
5. 今日任务页从学习包的训练配置中生成当天练习。
6. 用户完成今日任务后，本地记录进度和练习结果，并尝试同步后端。

所以当前设计更像：

```text
学习计划 = 用户选择的内容池
今日任务 = 从当前内容池里按天抽取的输出训练
```

这不是一个服务端每日调度系统。当前 `TodayTaskPage` 不调用 `/learning/today`，而是直接从离线学习包生成页面内容。

## 三、后端链路

### 1. 学习计划列表

入口：

- `GET /learning/my-units`
- 控制器：`apps/backend/src/modules/learning/learning.controller.ts`
- 服务：`apps/backend/src/modules/learning/learning.service.ts`

后端查询当前用户的 `UserSceneProgress`，并带出关联 `Scene`、分类、`TrainingTopic` 和数量统计。也就是说，只要某个 `Scene` 有了用户进度记录，它就会出现在学习计划里。

进度字段包括：

- `vocabLearned`
- `chunkMastered`
- `completedPracticeCount`
- `completedScriptCount`
- `readiness`
- `mastery`
- `completionPercent`

其中 `completedPracticeCount` 当前主要从已分析且分数超过阈值的 `PracticeSession` 推导。

### 2. 开始学习

入口：

- `POST /learning/units/:id/start`

行为：

- 校验学习包访问权限。
- 最多允许同时学习 3 个未完成单元。
- 对 `UserSceneProgress` 做 upsert。
- 已经存在进度的学习包可以继续学习。

这一步是“加入学习计划”的服务端依据。

### 3. 学习包详情和离线包

入口：

- `GET /learning/units/:id`
- `GET /learning/units/:id/offline-manifest`
- `GET /learning/units/:id/download-pack`
- `GET /learning/packs/check`
- `GET /learning/units/:id/download-delta`

今日任务需要的关键字段来自学习包详情中的：

```text
UnitDetail.trainingTopics[].metadata.outputTraining
```

其中 `outputTraining.pipeline` 是今日任务真正抽题的数据源。

### 4. 服务端今日任务接口

后端仍有：

- `GET /learning/today`
- `LearningService.getTodayTasks()`

它会选择最近更新且未完成的 `UserSceneProgress`，没有进行中进度时回退到第一个可解锁 `Scene`，然后生成词汇、句块、句型、开口练习、剧本挑战等任务。

但当前前端 `TodayTaskPage` 没有调用 `learningApi.getTodayTasks()`。因此这条接口目前更像旧版或备用设计，不能代表当前页面的主链路。

## 四、前端学习计划链路

主要文件：

- `apps/frontend/src/features/learning/pages/learning-plan-page.tsx`
- `apps/frontend/src/features/learning/components/my-learning-view.tsx`
- `apps/frontend/src/stores/learning.store.ts`
- `apps/frontend/src/lib/offline/learning.repository.ts`
- `apps/frontend/src/lib/offline/learning-pack.service.ts`

页面加载时：

1. `LearningPlanPage` 调用 `fetchMyLearning()` 和 `fetchDownloadedPacks()`。
2. `fetchMyLearning()` 先读 `learningRepository.getMyUnits()`。
3. `learningRepository.getMyUnits()` 优先返回本地 `my_learning_units`，本地没有时请求 `/learning/my-units`。
4. `MyLearningView` 根据 `downloaded_packs` 判断学习包是否已安装。
5. 未安装时，学习单元卡片会显示下载动作，并阻止进入学习单元详情。

用户加入学习包时：

1. `enrollUnit(unitId)` 调用 `learningRepository.enrollUnit()`。
2. repository 先把摘要写入本地 `my_learning_units`，并把 `my_unit:create` 放入 outbox。
3. 在线时调用 `/learning/units/:id/start`，成功后刷新 `/learning/my-units`。
4. store 将学习包加入下载队列。
5. 下载队列调用 `learningPackService.installUnit()`。
6. 安装完成后刷新 `downloaded_packs` 和 `myUnits`。

这保证学习计划页和今日任务页共享同一批本地学习包数据。

## 五、今日任务生成链路

主要文件：

- `apps/frontend/src/features/learning/pages/today-task-page.tsx`
- `apps/frontend/src/lib/offline/learning.repository.ts`
- `apps/frontend/src/lib/offline/practice.repository.ts`
- `apps/frontend/src/stores/warmup-session.store.ts`

### 1. 选择哪个学习包

`TodayTaskPage` 的选择顺序：

1. 如果 URL 有 `packId`，优先读取该学习包的本地详情。
2. 否则读取本地 `downloaded_packs` 和 `my_learning_units`。
3. 优先选择“我的学习中第一个已安装包”。
4. 如果没有匹配的我的学习包，则选择第一个已安装包。
5. 如果没有任何已安装包，显示空态，引导去学习计划选择学习包。

当前路由支持：

```text
/today
/today?packId=<sceneId>
```

不过现有学习计划卡片主要跳转到 `/learning/units/:id`，没有在学习计划页直接跳 `/today?packId=...`。

### 2. 从哪里抽题

今日任务遍历：

```text
unit.trainingTopics[].metadata.outputTraining.pipeline[]
```

支持的题型来自 `TYPE_META`：

- `chunk_substitution`：句块替换 / 词汇替换
- `vocab_drill`：词汇输出
- `vocab_sentence_building`：一词多句
- `pattern_drill`：句型操练
- `sentence_decomposition`：句子拆解

每个 pipeline item 会随机抽一个可练习条目。`runSeed` 默认来自当天日期的 hash，所以同一天初次进入会尽量稳定；点击换一批会改成随机种子重新生成。

### 3. 每日题量

今日任务读取用户偏好：

```text
usePreferencesStore((s) => s.dailyGoal)
```

生成策略：

1. 先遍历学习包里所有训练话题和 pipeline。
2. 按题型保证多样性，每种已有题型尽量先取 1 道。
3. 再随机补足到 `dailyGoal`。
4. 如果某些题型完全没有配置，会生成占位卡片，提示后台补题；占位卡片可以跳过并标记完成。

### 4. 今日进度恢复

今日任务把已完成题目的 step id 写入：

```text
daily_progress
```

字段包括：

- `date`
- `packId`
- `doneIds`

页面再次打开时会读取当天记录。如果 URL 指定了 `packId`，并且当天进度属于别的包，则不会恢复，避免跨包错乱。

### 5. 完成后记录

当 `doneCount === steps.length` 时：

1. 过滤掉占位卡片。
2. 调用 `practiceRepository.submitWarmupRecords()`。
3. 先写本地 `warmup_records`。
4. 尝试调用后端 `warmupRecordApi.save()` 和 `warmupRecordApi.assess()`。
5. 同步失败时写入 outbox，等待后续重试。
6. 调用 `markTodayActivity()`，写入 `daily_activity`。

## 六、离线缓存关系

当前联动依赖这些本地表：

| 表 | 作用 |
| --- | --- |
| `my_learning_units` | 学习计划页和今日任务页都用来判断“我的学习”。 |
| `downloaded_packs` | 判断学习包是否安装，今日任务只消费已安装包。 |
| `downloaded_unit_details` | 保存学习包详情和 topic 详情，是今日任务抽题的数据源。 |
| `warmup_records` | 保存今日任务完成后的练习明细与同步状态。 |
| `daily_progress` | 保存当天某个包已完成的 step id。 |
| `daily_activity` | 保存当天练习活跃次数，用于打卡统计。 |
| `sync_outbox` | 离线或同步失败时暂存创建/删除/练习记录等操作。 |

## 七、内容配置要求

如果一个学习包希望能出现在今日任务中，仅有 `TrainingTopic` 不够，还需要在 topic metadata 中配置输出训练：

```json
{
  "outputTraining": {
    "enabled": true,
    "pipeline": [
      {
        "id": "example-step",
        "type": "chunk_substitution",
        "chunk": "take care of",
        "chunkMeaning": "照顾",
        "items": [
          {
            "zh": "我需要照顾我的室友。",
            "answer": "I need to take care of my roommate."
          }
        ]
      }
    ]
  }
}
```

缺少 pipeline 的学习包仍然可以加入学习计划和下载，但今日任务会没有真实题目，或只显示占位题型。

## 八、当前边界和注意事项

1. `TodayTaskPage` 当前不使用 `/learning/today`，因此后端今日任务的词汇/句块限额逻辑不会影响当前页面。
2. 今日任务要求学习包已安装。本地没有 `downloaded_packs` 和 `downloaded_unit_details` 时，页面无法生成任务。
3. 学习计划允许最多 3 个未完成单元；今日任务默认只选其中一个包，不会自动混合多个学习包。
4. `packId` 可以强制指定今日任务来源，但学习计划页当前没有显式入口跳转到 `/today?packId=...`。
5. 同一天默认题目相对稳定，但“换一批题目”会重置本页进度并重新随机抽题。
6. 今日任务完成后保存的是 warmup 记录；它不会直接调用 `/learning/units/:id/progress` 更新 `UserSceneProgress`。
7. 服务端 `UserSceneProgress.completedPracticeCount` 主要基于正式 practice session 的分析结果推导，和今日任务 warmup 完成记录不是同一个进度来源。

## 九、后续可选优化

如果要让设计更一致，可以考虑：

1. 明确废弃或改造 `/learning/today`，避免服务端和前端存在两套今日任务模型。
2. 在学习计划卡片增加“今日练这个包”入口，跳转 `/today?packId=<unit.id>`。
3. 今日任务完成后同步影响 `UserSceneProgress`，或者在学习计划页单独展示 warmup 完成进度。
4. 将今日任务的选包策略抽到 repository 或 store，避免页面组件承担业务决策。
5. 后台学习包编辑页增加 `outputTraining.pipeline` 完整性检查，减少占位题型。

## 十、下一阶段设计决策

本节根据 2026-06-21 的产品讨论补充，作为后续实现依据。

### 1. 废弃 `/learning/today`

决策：废弃服务端 `/learning/today` 今日任务模型。

原因：

- 当前前端 `TodayTaskPage` 已经不调用 `/learning/today`。
- 服务端 `/learning/today` 生成的是词汇、句块、句型、practice、script 混合任务；前端实际生成的是 `outputTraining.pipeline` 的知识点输出练习。
- 两套模型会导致“今日任务到底是什么”无法解释，也会让后续学习算法和统计口径分裂。

推荐处理：

1. 前端移除 `learningApi.getTodayTasks()`、`TodayPlan`、`TodayTask` 等未使用类型。
2. 后端删除 `LearningController.getTodayTasks()` 和 `LearningService.getTodayTasks()`。
3. 如果担心旧客户端仍调用，可以短期保留接口但返回 `410 Gone` 或明确错误：`/learning/today has been deprecated; use offline daily task planner`。
4. 新的后端同步接口不要叫 `/learning/today`，避免语义混淆；建议使用 `/learning/daily-progress` 或 `/practice/warmup-progress`。

### 2. 不增加“今日练这个包”按钮

决策：学习计划页不增加“今日练这个包”按钮。

原因：

- 用户已经有底部导航的“今日任务”入口。
- 学习计划页的核心职责是管理学习包：开始、下载、更新、退出。
- 如果在每个学习包卡片加“今日练这个包”，会暗示用户每天要手动选择包，而当前目标更像“系统根据学习算法自动安排今日内容”。

保留能力：

- `/today?packId=<unit.id>` 可以作为调试或深链能力保留，但不作为主 UI 暴露。
- 今日任务默认选包策略应从页面组件抽出，后续交给学习算法决定，而不是靠按钮临时切换。

### 3. 今日任务和学习计划进度的关系

决策：今日任务完成后需要同步到后端，但不能混入 `PracticeSession` 口径。

需要区分三类练习：

| 类型 | 当前表 | 含义 | 是否影响学习计划话题进度 |
| --- | --- | --- | --- |
| 知识点练习 / warmup | `PracticeWarmupRecord` | 今日任务里的句块、词汇、句型、拆句等轻量输出练习。 | 应该影响“知识点练习进度”，但不应算作 VN 对话练习。 |
| VN / 正式话题练习 | `PracticeSession` | 进入 `/practice/session/:topicId` 的对话式练习和 AI 分析。 | 影响正式话题练习完成度。 |
| 剧本练习 | `StoryRecord` | VN 剧情 / script play 记录。 | 影响剧本完成度。 |

当前代码中，`UserSceneProgress.completedPracticeCount` 主要由 `PracticeSession` 分析结果推导。今日任务的 warmup 完成记录不会直接更新这个字段。

下一步建议新增独立统计口径：

- `warmupCompletedCount`：该学习包下已完成知识点练习数。
- `warmupDueCount`：今天应该复习/练习的知识点数。
- `warmupOverdueCount`：以前应该完成但未完成、累计到今天的知识点数。
- `warmupMastery`：知识点练习掌握度，可由题目通过率、最近复习状态、连续完成情况计算。

这些字段可以作为后端接口返回的统计结果，不一定全部落在 `UserSceneProgress` 上。`UserSceneProgress` 更适合继续保留宏观学习包进度。

### 4. 今日任务底部话题卡片进度条

决策：在 `/today` 底部“当前学习话题”卡片增加轻量进度条和状态色，而不是回到学习计划页增加入口按钮。

建议 UI：

- 每个话题卡片底部增加一条细进度条。
- 进度条前半或主体显示该话题知识点练习完成度。
- 卡片右侧或标题下方显示状态 badge。
- 不在卡片里塞太多文字，避免今日任务页变成学习计划页。

建议状态色：

| 状态 | 颜色 | 含义 |
| --- | --- | --- |
| 今日待练 | 蓝色 | 今天首次安排的新练习。 |
| 今日待复习 | 琥珀色 | 到了复习时间，需要巩固。 |
| 已逾期累积 | 红色 | 昨天或更早应完成但未完成，今天继续累积。 |
| 今日已完成 | 绿色 | 今天已完成该话题的今日任务。 |
| 长期掌握 | 紫色或中性灰 | 最近多次通过，短期无需复习。 |

话题卡片可展示：

- `todayNewCount`：今日新练数量。
- `todayReviewCount`：今日复习数量。
- `overdueCount`：逾期累计数量。
- `doneTodayCount` / `scheduledTodayCount`：今日完成比例。
- `topicWarmupProgress`：该 topic 的知识点练习整体完成度。

颜色优先级建议：

```text
overdueCount > 0       -> 红色
todayReviewCount > 0   -> 琥珀色
todayNewCount > 0      -> 蓝色
doneTodayCount 完成全部 -> 绿色
短期无需复习           -> 中性或紫色
```

### 5. 今日任务学习算法

目标：今日任务不只是随机抽题，而是能安排新练、复习、逾期累积，并且离线可用。

建议把今日任务拆成两层：

1. 候选池：从已安装学习包的 `outputTraining.pipeline` 展开出可练习 item。
2. 排程层：根据每个 item 的学习状态决定今天练什么。

每个可练习 item 需要有稳定 ID：

```text
packId:topicId:pipelineItemId:itemIndex
```

本地和后端都围绕这个 ID 记录状态，避免题目每天重排后无法对齐。

建议状态字段：

| 字段 | 含义 |
| --- | --- |
| `itemId` | 稳定题目 ID。 |
| `packId` / `topicId` | 所属学习包和话题。 |
| `type` | 题型，如 `chunk_substitution`、`vocab_drill`。 |
| `status` | `new`、`learning`、`review`、`mastered`。 |
| `dueDate` | 下次应练日期。 |
| `lastPracticedAt` | 最近练习时间。 |
| `streak` | 连续通过次数。 |
| `lapseCount` | 失败或遗忘次数。 |
| `easeFactor` | 间隔重复难度系数。 |
| `intervalDays` | 当前复习间隔。 |
| `doneToday` | 今天是否完成。 |
| `lastScore` | 最近一次表现，可用 `strong`、`ok`、`weak` 或 0-100。 |

排程策略：

1. 每天先取 `dueDate <= today` 的复习项。
2. 再取逾期项，逾期项不会消失，会累积到后续日期。
3. 如果复习量低于 `dailyGoal`，补充新题。
4. 新题优先覆盖不同题型，保持多样性。
5. 用户完成后根据表现更新 `dueDate`：
   - `strong`：间隔扩大，如 1 天 -> 3 天 -> 7 天 -> 14 天。
   - `ok`：小幅扩大或维持，如 1 天 -> 2 天 -> 4 天。
   - `weak`：明天继续复习，必要时重置 streak。
6. 如果今天没完成，未完成项保留原 `dueDate`，明天继续出现在 due/overdue 池中。

这套算法应该放在独立模块中，例如：

```text
apps/frontend/src/features/learning/daily-practice-planner.ts
apps/frontend/src/lib/offline/daily-practice.repository.ts
```

`TodayTaskPage` 只负责渲染 planner 的结果，不再自己决定所有选题策略。

### 6. 离线优先和后端同步

决策：今日任务学习状态必须本地先写、后端后同步。

本地建议新增或改造：

| 本地表 | 作用 |
| --- | --- |
| `daily_practice_items` | 每个稳定题目的复习状态、dueDate、streak、interval 等。 |
| `daily_practice_runs` | 每天生成的一次任务计划，记录 pack/topic、itemIds、状态。 |
| `daily_practice_attempts` | 每次作答记录，支持离线回放同步。 |

后端建议新增镜像表或等价模型：

| 后端模型 | 作用 |
| --- | --- |
| `UserWarmupItemProgress` | 用户对每个 warmup item 的长期状态。 |
| `UserDailyPracticeRun` | 某天生成的计划快照，便于后台统计和跨设备恢复。 |
| `UserDailyPracticeAttempt` | 每次作答记录，便于统计、评估、排查。 |

同步规则：

1. 练习完成时先写本地 `daily_practice_attempts` 和 item progress。
2. 同步任务写入 `sync_outbox`。
3. 在线时批量 upsert 到后端。
4. 后端使用 `clientAttemptId` 或 `runId + itemId + attemptIndex` 去重，保证重复同步不会重复计数。
5. 后端返回合并后的 item progress，前端再更新本地。
6. 离线多天时，未完成 due item 不丢失，按本地 dueDate 继续累积。

当前 `PracticeWarmupRecord` 可以继续保存一次练习的汇总和 AI 反馈，但不建议单独承担学习算法状态。它更像历史记录，不是调度状态。

注意：现有 `submitWarmupRecords()` 会调用 `save()` 和 `assess()`，而后端这两个接口都会创建 `PracticeWarmupRecord`。如果后续用 warmup record 做统计，需要先改成“一次完成只产生一条记录”，或在统计时按 run/attempt 去重。

### 7. 后台用户详情 Tab 设计

决策：后台用户详情 dialog 可以改为多 tab，用于管理和观察用户学习情况。

当前用户详情只展示基础资料、登录会话、账号、学习画像、练习/VN总数和 AI 用量。后续建议拆成：

| Tab | 内容 |
| --- | --- |
| 概览 | 头像、角色、会员、在线状态、输出等级、XP、积分。 |
| 学习包 | 用户加入的学习包、安装/同步状态、总体进度、最近练习时间。 |
| 知识点练习 | warmup 完成数、复习 due/overdue、平均分、最近记录、按学习包/话题聚合。 |
| VN 练习 | `PracticeSession` 数量、通过话题、平均分、最近分析结果。 |
| 剧本练习 | `StoryRecord` 完成、通过、回合数、XP。 |
| 账号与设备 | 登录会话、绑定账号、设备/IP。 |
| AI 用量 | 近 30 天 AI 调用、token、费用/额度。 |

后端建议不要把所有统计塞进 `GET /admin/users/:id`，否则 dialog 首次打开会很重。可以拆接口：

```text
GET /admin/users/:id
GET /admin/users/:id/learning-stats
GET /admin/users/:id/warmup-stats
GET /admin/users/:id/practice-stats
GET /admin/users/:id/story-stats
GET /admin/users/:id/ai-usage
```

也可以先做一个聚合接口：

```text
GET /admin/users/:id/learning-overview
```

返回学习包、warmup、VN、剧本的摘要；进入具体 tab 时再懒加载明细。

## 十一、歧义和冲突检查

目前需求整体不冲突，但有几处需要实现前明确：

1. “废弃 `/learning/today`”和“今日任务需要同步到后端”不冲突。废弃的是服务端生成今日任务的旧模型；新增的是同步今日任务状态和学习算法状态的接口。
2. “不需要今日练这个包按钮”和“今日任务底部话题进度条”不冲突。前者避免学习计划页增加入口；后者是在今日任务页解释当前安排和进度。
3. “不要和 practice session 混淆”和“学习计划要显示知识点练习完成度”不冲突。应新增 warmup/知识点进度口径，而不是复用 `PracticeSession.completedPracticeCount`。
4. “今日没完成明天继续累积”要求学习算法必须有稳定 item ID 和 dueDate，不能继续只靠当天随机抽题。
5. “离线也能流程”要求调度算法和状态更新首先在前端本地完成，后端只能作为同步镜像和跨设备恢复来源。

已确认的产品细节：

1. 今日任务排程范围做成系统设置：`single` 只从一个学习包排程；`mixed` 允许跨多个已安装学习包混合排程。
2. 复习无上限，新题受 `dailyGoal` 限制。也就是说逾期和到期复习不会因为目标数而消失，新题只在复习压力较低时补充。
3. warmup 通过标准直接使用题目组件返回的 `strong`、`ok`、`weak`。
4. 学习计划页总体 `completionPercent` 暂不混入 warmup 掌握度，避免把“知识点复习”和“学习包宏观完成度”绑死。warmup 进度先在 `/today` 话题卡片和后台用户详情里独立展示。
5. 跨设备合并时，后端保留所有 attempt 记录；长期 item progress 按“最高分 + 最新练习时间”合并。

## 十二、已落地实现

本轮实现采用“前端离线 planner + 后端同步镜像”的方案。

前端：

- 新增 `dailyPracticeRepository`，负责从已安装学习包展开候选题、读取系统设置、生成今日计划、更新本地 item progress、写入离线 attempt，并通过 outbox 重试同步。
- 新增 `useDailyPracticeStore`，让 `/today` 页面只消费计划、完成单题、提交当日结果，页面不再承担排程算法。
- `/today` 底部话题卡片增加 warmup 进度条和状态色：逾期、复习、新练、今日完成、短期掌握。
- SQLite 版本升级到 8，新增 `daily_practice_items`、`daily_practice_runs`、`daily_practice_attempts`。
- 前端移除旧 `learningApi.getTodayTasks()`、`TodayPlan`、`TodayTask`。
- 今日任务完成后仍写入本地 `daily_activity`，保留原打卡机制。
- 系统设置页新增“学习策略”分组，并用下拉菜单管理 `daily_practice_pack_scope`。

后端：

- `/learning/today` 改为 `410 Gone`，避免继续维护两套今日任务模型。
- 新增 `UserWarmupItemProgress`、`UserDailyPracticeRun`、`UserDailyPracticeAttempt` 及 migration。
- 新增 `/practice/daily-practice/settings`、`/practice/daily-practice/progress`、`/practice/daily-practice/complete`。
- 后端同步时按 `clientAttemptId` 去重；attempt 全量保留，item progress 按最高分和最新时间合并。
- 后台用户详情新增 `GET /admin/users/:id/learning-overview`，聚合学习包、知识点 warmup、VN 练习、剧本练习统计。
- 用户管理详情 dialog 增加学习相关 tabs，用于查看学习包、知识点、VN、剧本统计。

仍需后续加强：

1. 后台用户学习统计目前是摘要口径，后续可以按 tab 懒加载明细列表。
2. warmup 对学习计划页宏观进度是否产生权重影响，仍建议等使用一段时间后再决定，避免早期统计口径频繁变化。
